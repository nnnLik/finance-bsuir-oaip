# Учёт доходов и расходов (Delphi VCL, SQLite)

Самодостаточное описание приложения для человека, у которого нет доступа к исходному коду.

## Что это за приложение

Десктопное Win32-приложение на Delphi VCL для личного учета финансов:
- ведение доходов и расходов;
- работа с несколькими счетами;
- автосохранение всех действий в SQLite;
- анализ данных по выбранному счету или по всем счетам;
- импорт и экспорт операций в CSV;
- управление категориями доходов и расходов.

Приложение запускается в графическом режиме, работает как single-instance (второй экземпляр не запускается).

## Основные возможности

- **Операции**: добавить, изменить, удалить запись (дата, тип, категория, описание, сумма).
- **Фильтрация на главной**: фильтр по категории (без поиска по тексту).
- **Счета**:
  - есть активный счет;
  - можно создавать дополнительные счета;
  - можно удалять счета (с подтверждением, удаляются и его транзакции);
  - последний счет удалить нельзя.
- **Категории**:
  - отдельные списки для доходов и расходов;
  - можно добавлять и удалять категории;
  - при создании/редактировании операции список категорий зависит от типа (доход/расход).
- **Аналитика (вкладки)**:
  - обзор;
  - расходы по категориям;
  - доходы по категориям;
  - тренды по месяцам.
- **Область аналитики**:
  - только активный счет;
  - все счета.
- **CSV**:
  - экспорт операций текущего активного счета в файл;
  - импорт из CSV с валидацией строк и подсчетом импортированных/пропущенных.

## Как устроен интерфейс

Главное окно состоит из 3 страниц (навигация иконками в левом сайдбаре):

1. **Главная**
   - таблица операций;
   - фильтр по категории;
   - кнопка добавления операции;
   - действия редактирования и удаления через иконки в таблице.

2. **Аналитика**
   - переключатель области данных: активный счет / все счета;
   - вкладки:
     - **Обзор**: сумма доходов, сумма расходов, баланс, количество операций;
     - **Расходы**: распределение по категориям;
     - **Доходы**: распределение по категориям;
     - **Тренды**: динамика доходов и расходов по месяцам.

3. **Настройки**
   - экспорт CSV;
   - импорт CSV;
   - создание/удаление счетов;
   - добавление/удаление категорий доходов;
   - добавление/удаление категорий расходов.

В верхней панели отображаются:
- текущий баланс активного счета;
- селектор активного счета.

## Логика работы (бизнес-правила)

- Все действия сохраняются сразу в БД (кнопки “Сохранить” нет).
- Баланс в верхней панели всегда рассчитывается для активного счета.
- При переключении счета:
  - главная таблица и аналитика перезагружаются;
  - при отсутствии операций отображается пустая таблица (без “призрачных” строк).
- При удалении счета показывается предупреждение о каскадном удалении его операций.
- При создании/редактировании операции:
  - сумма должна быть положительной;
  - категория обязательна;
  - для расходов показываются только расходные категории, для доходов — только доходные.
- При импорте CSV:
  - некорректные строки пропускаются;
  - отсутствующий счет создается автоматически;
  - при необходимости создается категория из CSV.

## Хранение данных

### Основное хранилище

SQLite база `finance.db` рядом с `Finance.exe`.

Таблицы:
- `accounts` — счета;
- `categories` — категории доходов/расходов;
- `transactions` — операции;
- `app_settings` — служебные настройки (активный счет, область аналитики).

### Первый запуск

При первом запуске рядом с `Finance.exe` создается `finance.db`, таблицы, счет по умолчанию и стартовые категории. Импорт старых файлов или миграция из прежних форматов не выполняются.

## Формат CSV

Заголовок:

`date_str,is_income,category,description,amount,account_name`

Поля:
- `date_str` — дата в формате `dd.mm.yyyy`;
- `is_income` — `1` (доход) или `0` (расход);
- `category` — название категории;
- `description` — описание;
- `amount` — положительное число;
- `account_name` — имя счета.

Пример:

```csv
date_str,is_income,category,description,amount,account_name
06.04.2026,1,Зарплата,Апрель,1500.00,Основной
07.04.2026,0,Продукты,Супермаркет,58.40,Основной
```

## Архитектура проекта

Проект разделен на уровни:

- **App**
  - `Finance.dpr` — точка входа, single-instance (mutex), запуск главной формы.

- **Core**
  - `Finance.Types.pas` — базовые типы (`TTransactionRec`, `TAccountRec`, узлы списка);
  - `Finance.Strings.pas` — централизованные текстовые константы;
  - `Finance.ChartTheme.pas` — палитра и шрифт для графиков аналитики;
  - `Finance.DateUtils.pas` — разбор/формат даты;
  - `Finance.Categories.pas` — фасад работы с категориями.

- **Data**
  - `Finance.Db.pas` — подключение к SQLite и схема БД;
  - `Finance.Repository.pas` — CRUD и сценарии работы с БД/CSV;
  - `Finance.TransactionList.pas` — собственный односвязный список и операции над ним.

- **UI / логика экранов**
  - `Finance.AnalyticsTabsService.pas` — агрегаты для графиков и вкладок аналитики;
  - `UnitMainShell.pas` — оболочка, навигация, выбор счета, баланс;
  - `UnitMain.pas` — главная таблица операций;
  - `UnitTransactionEditor.pas` — модальное окно создания/редактирования записи;
  - `UnitAnalytics.pas` + `UnitAnalyticsTab*.pas` — аналитика по вкладкам;
  - `UnitSettings.pas` — настройки, CSV, счета, категории.

## Жизненный цикл данных

1. Пользователь выполняет действие в UI (добавление/редактирование/удаление).
2. Форма передает данные в `Repository`.
3. `Repository` пишет изменения в SQLite.
4. UI обновляет таблицы/сводки/баланс.

## Технические детали

- Платформа: Win32, VCL.
- Работа с БД: FireDAC + SQLite (`sqlite3.dll` через стандартный стек Delphi).
- Графики аналитики: рисование на `TPaintBox` (без зависимости от TeeChart).
- Стиль интерфейса: VCL style (константа `APP_STYLE_NAME` в `Finance.Strings.pas`, по умолчанию `Glossy`).

## Соответствие обязательным требованиям курсовой

Ниже — привязка формулировок из задания к фактической реализации в репозитории.

### 1. Динамические структуры данных

| Что используется | Где в коде | Комментарий |
|------------------|------------|-------------|
| **Односвязный список** узлов с операциями вставки, обхода, освобождения | `Finance.Types.pas` (`PListNode`, `TListNode`), `Finance.TransactionList.pas` | Узлы выделяются через `New`/`Dispose`, связь `Next`. Список — основная структура для операций в памяти при загрузке из БД и расчётах. |
| **Динамические массивы** | `Finance.Types.pas` (`TAccountArray = array of TAccountRec`), `Finance.AnalyticsTabsService.pas` (массивы точек графиков и трендов) | `SetLength`, изменяемая длина. |

Использование `TStrings` у `TComboBox` — стандартный интерфейс VCL для отображения списков; предметная модель данных при этом хранится в списке и в SQLite, а не в коллекциях вроде `TList<T>` из Generics.

### 2. Работа с файлами средствами языка (`TextFile`)

Запрещённые для «зачёта по файлам» обёртки вида `TMemo.LoadFromFile` / `SaveToFile` в проекте **не используются** (поиск по исходникам пустой).

| Механизм языка | Назначение | Модуль / процедура |
|----------------|------------|-------------------|
| **`TextFile`** (`AssignFile`, `Reset`/`Rewrite`, `ReadLn`/`WriteLn`) | Импорт и экспорт CSV | `Finance.Repository.pas`: `RepoExportCsv`, `RepoImportCsv`, вспомогательные `CsvWriteEscaped`, разбор строки `CsvSplit` |

Основное хранилище — SQLite (`finance.db` через FireDAC). Файловый обмен через средства языка реализован только для **CSV** (импорт/экспорт в настройках). Если в задании явно требуются также `file` / `file of <тип>`, их можно добавить отдельным небольшим модулем под тестовый обмен без возврата к старым форматам данных.

### 3. Графический пользовательский интерфейс

- Точка входа: `Finance.dpr`, директива `{$APPTYPE GUI}`, использование `Vcl.Forms` и форм.
- Главное окно и страницы: `UnitMainShell.pas`, `UnitMain.pas`, `UnitAnalytics.pas`, `UnitSettings.pas`, модальное окно записи `UnitTransactionEditor.pas`.

Консольного режима нет.

### 4. Не менее 5 собственных алгоритмов (реализованных в коде)

Для пояснительной записки можно оформить таблицу «не менее 5» из списка ниже (все процедуры/функции с нетривиальной логикой, написанной в проекте):

| № | Алгоритм (суть) | Модуль | Имя |
|---|------------------|--------|-----|
| 1 | Вставка в конец односвязного списка | `Finance.TransactionList.pas` | `ListAddToEnd` |
| 2 | Полное освобождение списка | то же | `ListDispose` |
| 3 | Вычисление баланса одним проходом по списку | то же | `ListCalcBalance` |
| 4 | Разбор CSV-строки с учётом кавычек и запятых | `Finance.Repository.pas` | `CsvSplit` |
| 5 | Агрегирование сумм по категориям (доход/расход) | `Finance.AnalyticsTabsService.pas` | `BuildCategories` (через `BuildExpenseCategories` / `BuildIncomeCategories`) |
| 6 | Сортировка категорий по убыванию суммы | то же | `SortCategoriesByValueDesc` |
| 7 | Группировка по месяцам и сортировка периодов | то же | `BuildMonthlyTrends`, `SortMonthsAsc` |
| 8 | Поиск узла списка по индексу строки с учётом фильтра | `UnitMain.pas` | `GetNodeByDisplayIndex` (совместно с `MatchesFilters`) |
| 9 | Импорт CSV: разбор строк, валидация, создание счёта/категории при необходимости | `Finance.Repository.pas` | `RepoImportCsv` |

Итого требование «≥ 5 алгоритмов» выполняется с запасом.

## Сборка и запуск

- Проект: `src/App/Finance.dproj`.
- Конфигурация: Debug/Release Win32.
- Если сборка падает с ошибкой блокировки `Finance.exe`, закройте запущенный процесс и пересоберите проект.

## Ограничения текущей версии

- Экспорт из интерфейса выполняется только по активному счету.
- Удаление категории не проверяет, используется ли она уже в существующих транзакциях.
- Валидация CSV при импорте базовая (некорректные строки пропускаются).
