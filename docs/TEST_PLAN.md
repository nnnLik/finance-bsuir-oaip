# План тестирования Finance

> **Устаревшие разделы:** сценарии ниже (поиск, `finance.dat`, `categories.txt`, сохранение при закрытии) относятся к старой версии. Актуальное поведение — в `README.md` (SQLite, автосохранение, CSV через настройки, оболочка `UnitMainShell`).

## 1. Цель

Подтвердить корректность пользовательских сценариев приложения учета доходов и расходов и проверить устойчивость к ошибочным данным и нештатным ситуациям.

## 2. Границы тестирования

### Включено в покрытие
- Главная: добавление, редактирование, удаление, фильтр по категории, таблица операций.
- Счета: переключение, создание/удаление в настройках.
- Аналитика: вкладки, область «активный счёт / все счета».
- Персистентность: SQLite `finance.db`, автосохранение при действиях.
- CSV: импорт и экспорт через экран настроек (`TextFile` в коде).

### Вне покрытия
- Визуальная pixel-perfect проверка иконок и оформления.
- Производительность на слабом железе за пределами базового стресс-прогона.

## 3. Объекты тестирования

- `finance/src/UI/Forms/UnitMainShell.pas`, `UnitMain.pas`, `UnitAnalytics.pas`, `UnitSettings.pas`, `UnitTransactionEditor.pas`
- `finance/src/Data/Finance.Repository.pas`, `Finance.Db.pas`, `Finance.TransactionList.pas`
- `finance/src/Core/Finance.Categories.pas`, `Finance.DateUtils.pas`

## 4. Тестовая среда

- ОС: Windows 10/11.
- Сборка: Win32 (`finance/src/App/Finance.dproj`).
- Чистый старт при необходимости: удалить `finance.db` в каталоге exe (создастся заново).

## 5. Единый набор тестовых данных

### Категории
- Доход: `Зарплата`, `Фриланс`.
- Расход: `Продукты`, `Транспорт`, `Жилье`.
- Пользовательские: `Кэшбэк` (доход), `Кофе` (расход).

### Суммы
- Валидные: `1000`, `10.50`, `10,50`, `9999999.99`.
- Невалидные: `0`, `-1`, `abc`, `10..2`, `10,2,3`.

### Даты
- Валидные: текущая дата, даты из разных месяцев и годов, край месяца.
- Для файла: вручную подготовленные записи с невалидной датой-строкой (проверка аналитики по месяцам).

### Описания
- Пустое.
- Короткое (`Обед`).
- Длинное (на границе допустимой длины поля).

## 6. Позитивные тесты (блок P)

### P1. CRUD операций

#### P1.1 Добавление дохода
- Предусловие: список пуст.
- Шаги: открыть диалог -> выбрать тип "Доход" -> заполнить валидные поля -> `OK`.
- Ожидаемо: новая строка в таблице, баланс увеличен на сумму, счетчик операций +1.

#### P1.2 Добавление расхода
- Предусловие: в списке минимум одна запись.
- Шаги: добавить запись типа "Расход" с валидными полями.
- Ожидаемо: строка добавлена, баланс уменьшен, аналитика обновляется при открытой вкладке.

#### P1.3 Редактирование записи
- Предусловие: в списке минимум две записи.
- Шаги: нажать "редактировать" у строки -> изменить категорию/сумму -> `OK`.
- Ожидаемо: изменены данные нужной строки, баланс и агрегаты пересчитаны.

#### P1.4 Удаление записи
- Предусловие: в списке минимум одна запись.
- Шаги: нажать "удалить" у строки.
- Ожидаемо: строка исчезла, итоговые суммы и баланс пересчитаны корректно.

### P2. Валидный ввод в диалоге

#### P2.1 Сумма с точкой
- Шаги: создать запись с `10.50`.
- Ожидаемо: запись создается, сумма интерпретируется как 10.50.

#### P2.2 Сумма с запятой
- Шаги: создать запись с `10,50`.
- Ожидаемо: запись создается, сумма интерпретируется как 10.50.

#### P2.3 Пользовательская категория
- Шаги: ввести новую категорию (`Кэшбэк`/`Кофе`) и сохранить запись.
- Ожидаемо: категория доступна в соответствующем списке при следующем открытии диалога.

### P3. Фильтры и поиск

#### P3.1 Поиск без учета регистра
- Шаги: ввести поисковую строку в другом регистре, чем в данных.
- Ожидаемо: отображаются совпадающие записи по категории или описанию.

#### P3.2 Комбинированная фильтрация
- Шаги: выбрать категорию и задать поиск.
- Ожидаемо: остаются только записи, удовлетворяющие обоим условиям.

#### P3.3 Сброс к "Все"
- Шаги: после фильтра выбрать "Все".
- Ожидаемо: возвращается полный набор строк.

### P4. Аналитика

#### P4.1 Общие показатели
- Шаги: подготовить набор доходов и расходов с известными суммами.
- Ожидаемо: доходы, расходы, баланс и число операций равны ручному расчету.

#### P4.2 По категориям
- Шаги: добавить несколько записей в разные категории с разными суммами.
- Ожидаемо: блоки доходов/расходов по категориям отсортированы по убыванию суммы.

#### P4.3 По месяцам
- Шаги: добавить операции в 7+ разных месяцев.
- Ожидаемо: отображаются только 6 последних месяцев и правильные суммы по каждому.

### P5. Файлы и персистентность

#### P5.1 Сохранение и загрузка `finance.dat`
- Шаги: сохранить список -> закрыть/перезапустить -> загрузить.
- Ожидаемо: состав записей полностью совпадает с сохраненным.

#### P5.2 Закрытие с "Сохранить"
- Шаги: изменить данные -> закрыть приложение -> выбрать "Сохранить" -> открыть снова.
- Ожидаемо: изменения присутствуют.

#### P5.3 Сохранение категорий в `categories.txt`
- Шаги: добавить пользовательские категории -> перезапустить приложение.
- Ожидаемо: категории подхватываются без потерь.

## 7. Негативные тесты (блок N)

### N1. Невалидный ввод в диалоге

#### N1.1 Нулевая сумма
- Шаги: ввести `0` и нажать `OK`.
- Ожидаемо: запись не создается, пользователь получает сообщение о некорректном вводе.

#### N1.2 Отрицательная сумма
- Шаги: ввести `-1` и нажать `OK`.
- Ожидаемо: запись не создается.

#### N1.3 Нечисловое значение
- Шаги: поочередно ввести `abc`, `10..2`, `10,2,3`.
- Ожидаемо: запись не создается.

#### N1.4 Отмена диалога
- Шаги: заполнить поля -> нажать `Отмена`.
- Ожидаемо: список не изменяется.

### N2. Фильтрация и индексация

#### N2.1 Редактирование в отфильтрованном виде
- Шаги: отфильтровать список -> отредактировать видимую строку.
- Ожидаемо: изменяется именно выбранная запись, соседние не затрагиваются.

#### N2.2 Удаление в отфильтрованном виде
- Шаги: отфильтровать список -> удалить видимую строку.
- Ожидаемо: удаляется правильная запись, без смещения логики выбора.

#### N2.3 Пустой результат поиска
- Шаги: задать строку, которой нет в данных.
- Ожидаемо: таблица показывает 0 строк, ошибок и артефактов нет.

### N3. Файлы операций

#### N3.1 Отсутствующий `finance.dat`
- Шаги: удалить/переименовать файл -> выполнить загрузку.
- Ожидаемо: приложение не падает, показывает предсказуемое сообщение/пустое состояние.

#### N3.2 Поврежденный бинарный файл
- Шаги: подменить `finance.dat` произвольным бинарным содержимым -> загрузить.
- Ожидаемо: приложение не падает; текущие данные остаются консистентными.

#### N3.3 Пустой файл
- Шаги: создать пустой `finance.dat` -> загрузить.
- Ожидаемо: открывается пустой список без ошибок.

### N4. Файлы категорий

#### N4.1 Отсутствующий `categories.txt`
- Шаги: удалить файл и перезапустить.
- Ожидаемо: используются дефолтные категории.

#### N4.2 Мусорные строки/секции
- Шаги: добавить в `categories.txt` неизвестные заголовки и произвольный текст.
- Ожидаемо: приложение игнорирует невалидные строки и остается стабильным.

#### N4.3 Дубликаты категорий в разных регистрах
- Шаги: попытаться добавить категорию, уже существующую в ином регистре.
- Ожидаемо: дубликат не появляется.

### N5. Закрытие приложения

#### N5.1 Закрытие с "Не сохранять"
- Шаги: изменить данные -> закрыть -> выбрать "Не сохранять" -> запустить снова.
- Ожидаемо: новые изменения отсутствуют.

#### N5.2 Закрытие с "Отмена"
- Шаги: изменить данные -> закрыть -> выбрать "Отмена".
- Ожидаемо: приложение остается открытым, данные в текущей сессии сохранены.

## 8. Нефункциональные проверки

### NF1. Базовый объем
- Шаги: добавить 500+ записей.
- Ожидаемо: фильтрация, прокрутка и обновление аналитики выполняются без критических задержек.

### NF2. Циклическая устойчивость
- Шаги: 10-20 циклов `добавить -> удалить -> сохранить -> загрузить`.
- Ожидаемо: без сбоев, утечек и деградации поведения.

## 9. Порядок прогона

- Фаза 1: P1 -> P5.
- Фаза 2: N1 -> N5.
- Фаза 3: NF1 -> NF2.

## 10. Критерии завершения тестирования

- Все кейсы из блоков P и N имеют статус (`PASS`/`FAIL`/`BLOCKED`).
- Для каждого `FAIL` оформлен дефект с шагами воспроизведения.
- Покрыты все функциональные требования из `finance/docs/FUNCTIONAL.md`.
